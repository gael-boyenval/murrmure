# Plan review — Testability and safety deep audit

**Reviewer:** test and quality architecture  
**Date:** 2026-07-09  
**Plan reviewed:** [2026-07-09-mcp-reliability-plan.md](./2026-07-09-mcp-reliability-plan.md)

## Summary verdict

**PASS WITH HIGH-RISK GAPS (not merge-safe yet)**

The architecture direction is good, but the plan is not yet sufficiently executable from a test and safety standpoint. Current code and tests still reflect fat-CLI MCP behavior, while the plan requires a thin bin + hub-owned runtime model. Several acceptance statements are manual-only and currently not proxied by CI checks.

Top findings:

1. **Doctor checks are inverted vs target architecture.** `scanMcpConfig` currently accepts `murrmure mcp` as canonical and flags `murrmure-mcp` as legacy.
2. **Global MCP default cannot be validated.** Doctor scans project-local `.cursor/mcp.json` only; `~/.cursor/mcp.json` is ignored.
3. **Schema coverage is incomplete.** Only emit-event schema presence is tested; protocol tools in Phase 1 minimum set are not schema-gated.
4. **Fat CLI deletion is high blast radius.** Runtime, command wiring, build entrypoints, templates, tests, and docs are still tied to fat behavior.
5. **No thin-bin subprocess test harness exists today.** There is no Cursor-independent stdio integration test with MCP SDK client.
6. **Breaking-change rollback is unspecified.** There is no explicit deprecation window, fallback gate, or release-phase migration plan.
7. **Safety checks for revoked token and wrong-space token are not fully automated.**

## Thin bin integration test spec

### Goal

Validate thin-bin MCP behavior without Cursor by driving stdio with an MCP SDK client.

### Test harness

- Start hub fixture with random port.
- Write `~/.murrmure/hubs/shared.json` in isolated test home.
- Mint token grant for known space.
- Spawn `murrmure-mcp` child process with controlled env.
- Use MCP SDK client transport to run initialize/list/call flows.

### Fixtures

- `validDiscovery` with reachable endpoint
- `missingDiscovery` file case
- `invalidDiscovery` malformed JSON and missing endpoint cases
- `validToken`, `revokedToken`, and `missingToken` cases
- hub fixture with at least one flow/tool in catalog

### Concrete cases and assertions

| Case | Fixture | Assertions |
|------|------|------|
| Startup success | valid discovery + valid token | process starts and initialize succeeds |
| List parity | valid discovery/token | `tools/list` names match hub catalog names |
| Schema passthrough | valid discovery/token | `resolve_step` schema contains required fields |
| Call success | valid discovery/token | `murrmure_space_status` call returns expected payload |
| Validation passthrough | valid discovery/token + bad args | `{}` call returns hub validation error text |
| Authorization failure | revoked token | tool call returns denied error path (not generic crash) |
| Missing discovery | missing file | non-zero exit + actionable stderr mentioning discovery |
| Malformed discovery | invalid file | non-zero exit + parse/config error |
| Hub down | endpoint unreachable | timeout-safe, clear connectivity message |
| Multi-space denial | token for A, target B | denied with token-space error |
| Log safety | any valid run | token not printed to stdout/stderr |

### Wake coverage after fat deletion

Wake behavior should be validated at hub route level:

- handshake register principal
- invoke action dispatch
- handshake replay includes `murrmure/control.invoke_action`

This replaces CLI `control-session` behavior tests.

## Recommended plan amendments

1. Add **CI-verifiable** and **manual sign-off** subsections under each phase and final success criteria.
2. Add a hard **Phase 2 cutover gate**: thin-bin tests green, doctor canonical logic flipped, fat runtime removed in the same milestone.
3. Add a **rollback/deprecation strategy** (one release warning window) before hard deleting fat command paths.
4. Make Phase 4 doctor checks normative by code and include global config scan requirement.
5. Define thin-bin subprocess integration suite as a required deliverable in the plan itself.
6. Add explicit security criteria: no literal token in tracked snippets/templates/docs, and no token logging.
7. Add a multi-space safety criterion tied to ISSUE-07 fixture.
8. Define wake behavior ownership tests post-deletion (hub handshake replay, no CLI coupling).
9. Add docs-proof gates for `apps/docs/guide/agents-mcp.md` and `apps/docs/guide/cli.md` to prevent reintroduction of fat snippet guidance.
10. Add PR checklist items for phase DoD: tests updated, docs updated, migration notes, rollback notes.

---

## Final assessment

The plan is strong on architecture intent but still under-specified for test-enforced migration safety. It should proceed only with the CI/manual split and thin-bin/doctor/deletion gates above added as explicit deliverables.
